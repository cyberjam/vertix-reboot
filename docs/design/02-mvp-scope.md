# MVP 범위 정의 (요구사항 8)

> 신뢰도 태그는 [README](./README.md) 참조. 본 문서의 대부분은 **[설계 결정]** 이다.
> 분석 근거는 [01-game-analysis.md](./01-game-analysis.md) 참조.

## 0. MVP의 목적

**"Vertix.io의 핵심 게임성이 브라우저에서 실제로 작동하는가"를 가장 빠르게 검증**하는 것.
화려함보다 **핵심 루프(이동→교전→킬→리스폰→클래스 스위칭)의 손맛(feel)** 과
**멀티플레이어 동기화**가 동작하는 것을 최우선으로 한다.

> 원칙: *재미의 80%는 클래스 20%·모드 20%에서 나온다.* MVP는 그 20%만 정확히 만든다.

---

## 1. MVP 포함 범위 (In Scope)

### 1.1 핵심 게임 루프 (필수)
- [x] 4분 라운드 → 결과 화면 → 재시작 루프
- [x] 리스폰 시 클래스 재선택
- [x] 좁은 미로 맵에서의 탑다운 교전

### 1.2 클래스 (10개 중 **3개**로 시작)
원본의 트레이드오프 축을 대표하도록 선별:

| 클래스 | 역할 | 선정 이유 |
|--------|------|-----------|
| **Triggerman** | 밸런스/기준점 | 히트스캔 자동무기의 표준, 입문용 |
| **Hunter** | 원거리·유리몸 | 저격(단발 히트스캔) + 보조무기로 무기 다양성 검증 |
| **Vince (Shotgun)** | 근접 | 산탄(다중 레이) + 미로 코너 플레이 검증 |

> 이 3개로 **히트스캔 / 단발 / 산탄** 발사 모델과 **고/중/저 기동·체력** 트레이드오프를 모두 커버.
> 투사체(Rocketeer/Nademan)·지속(Arsonist)·조크(Duck)는 MVP 이후.

### 1.3 무기 발사 모델 (MVP는 **히트스캔 계열만**)
- [x] 자동 히트스캔 (머신건)
- [x] 단발 히트스캔 (스나이퍼)
- [x] 다중 히트스캔 (샷건 = 펠릿 N개 레이)
- [x] 탄창/재장전 (예비탄 무한 규칙)
- [ ] 투사체·스플래시·화염방사 — **MVP 제외** (Phase 2)

### 1.4 게임 모드 (**1~2개**)
- [x] **Free For All (FFA)** — 1500점(=15킬) 선취. 가장 단순하고 핵심 루프 검증에 최적.
- [x] (여력 시) **Team Deathmatch** — 팀 동기화·스폰 로직 검증.
- [ ] Hardpoint / Lootcrate / Boss Hunt / War 계열 — Phase 2+

### 1.5 맵
- [x] **미로형 맵 1개** (좁은 통로 + 코너 + 직선 저격 라인 + 힐팩 2~4개 + 스폰 다수)
- [x] 타일 그리드 기반 충돌

### 1.6 시스템
- [x] **리스폰 시스템** — 딜레이 + 안전 스폰 + 짧은 무적
- [x] **힐팩 시스템** — 고정 위치 픽업, 쿨다운 재스폰, 정량 회복
- [x] **점수 시스템** — 킬 100점, 서버 권위 집계, 라운드 종료 판정
- [x] **멀티플레이어** — 권위 서버 + WebSocket, 클라이언트 예측/보간
- [x] **기본 HUD** — 체력, 탄창, 점수판(킬 리더보드), 라운드 타이머
- [x] **닉네임 입력**(게스트) — 계정/로그인 불필요

## 2. MVP 제외 범위 (Out of Scope)

명시적으로 **Phase 2 이후**로 미룬다.

- 나머지 7개 클래스, 투사체/화염/AoE 무기
- 7개 모드(Hardpoint, Lootcrate, Boss Hunt, Sniper/Rocket/Pyro War 등)
- 계정/로그인, 영구 진행도, **코스메틱·스킨·드랍 보상**
- 매치메이킹/로비, 모드 투표 UI
- 모바일 터치 컨트롤, 사운드/음악 풀세트(최소 SFX만)
- 안티치트, 랭킹/리더보드 영속화, 채팅
- 다중 서버/샤딩, 관전 모드

## 3. 기술 스택 제안 **[과거 권고안 — 아래 확정안으로 대체됨]**

> ⚠️ **확정 스택은 Next.js + TypeScript + Phaser 3 + Colyseus + Supabase 이다.**
> 상세는 [03-technical-design.md](./03-technical-design.md) 참조. 이 절(아래)은 초기 권고안 기록으로만 보존한다.

### 3.1 아키텍처: **서버 권위(Authoritative Server)**
.io 슈터의 표준. 치트 방지 + 동기화 신뢰성. 클라이언트는 예측/렌더만 담당.

```
[Browser Client]  ←— WebSocket(바이너리) —→  [Game Server (Node)]
  - 입력 캡처                                   - 권위 시뮬레이션 (tick 30~60Hz)
  - 클라 예측/보간                              - 충돌·데미지·점수 판정
  - 렌더링(Canvas/WebGL)                        - 스냅샷 브로드캐스트(델타)
```

### 3.2 후보 스택

| 레이어 | 1순위 권고 | 대안 | 비고 |
|--------|-----------|------|------|
| 언어 | **TypeScript** (클라+서버 공유) | JS | 클라/서버 타입·로직 공유 용이 |
| 렌더링 | **PixiJS (WebGL 2D)** | Canvas2D, Three.js | 탑다운 2D 스프라이트에 최적. 3/4 시점은 스프라이트로 표현 |
| 네트워킹 | **WebSocket** (`ws` 또는 `uWebSockets.js`) | WebTransport | MVP는 WS로 충분. 바이너리 직렬화 |
| 서버 런타임 | **Node.js** | Bun, Go(성능 필요 시) | TS 공유 위해 Node 우선 |
| 직렬화 | 바이너리(스키마 기반, 예: flatbuffers/수동) | JSON(초기 프로토타입) | 대역폭 절감 |
| 빌드 | **Vite** | esbuild | 빠른 DX |
| 모노레포 | pnpm workspaces (`client`/`server`/`shared`) | — | `shared`에 클래스·무기·맵 데이터 + 시뮬 로직 공유 |

### 3.3 권장 디렉터리 구조 (착수 시)

```
vertix-reboot/
├─ docs/design/          ← (현재) 설계 문서
├─ packages/
│  ├─ shared/            ← 클래스/무기/맵 데이터, 결정론적 시뮬 로직, 프로토콜 타입
│  ├─ server/            ← 권위 게임 루프, 룸 관리, WS 핸들러
│  └─ client/            ← 렌더링, 입력, 예측/보간, HUD
└─ ...
```

### 3.4 핵심 기술 리스크 (MVP에서 반드시 검증)

1. **네트워크 동기화 손맛** — 지연/지터 하에서 사격 적중감 (lag compensation, 클라 예측).
2. **결정론적 충돌** — 서버/클라 동일 충돌 결과(특히 히트스캔 레이 vs 타일/벽).
3. **탑다운 가독성** — 좁은 통로에서 시야/벽 가림 처리, 캐릭터 식별성.
4. **틱레이트 vs 인원** — 30~60Hz로 동시 N명(목표 8~16명) 처리 가능 여부.

## 4. 마일스톤 (제안)

| 단계 | 목표 | 산출물 |
|------|------|--------|
| **M0** | 설계 확정 | 본 문서 리뷰·합의, 수치 캘리브레이션 |
| **M1** | 싱글 클라 프로토타입 | Triggerman 1종, 1맵, 이동·사격·충돌(로컬) |
| **M2** | 멀티플레이어 코어 | 권위 서버 + WS, 2명 동기화, 예측/보간 |
| **M3** | 클래스 3종 + 리스폰/힐팩 | 클래스 스위칭, 리스폰, 힐팩 픽업 |
| **M4** | FFA 모드 + HUD + 점수 | 라운드 루프 완성, 리더보드, 결과 화면 |
| **M5** | 폴리시 + 플레이테스트 | 밸런스 튜닝, 손맛 보정, TDM(여력 시) |

**MVP 완료 정의(Definition of Done):** *낯선 사람 여러 명이 한 브라우저 링크로 접속해,
3개 클래스로 4분 FFA를 끝까지 즐겁게 플레이하고, 점수/승패가 정확히 집계된다.*

## 5. 다음 단계 (의사결정 필요 항목)

착수 전 확정이 필요한 사항:

1. **원본 자료 확보** — 무기 수치(연사·재장전·사거리) 캘리브레이션용 영상/데이터.
2. **기술 스택 승인** — 위 3.2안 채택 여부 (특히 PixiJS vs Three.js, Node vs Bun/Go).
3. **목표 동시 인원** — 8명 / 16명 / 그 이상? (틱레이트·대역폭 설계에 영향)
4. **MVP 모드** — FFA만 vs FFA+TDM.
5. **아트 방향** — 원본 스프라이트 재현 vs 신규 아트(저작권 고려).
