# Vertix.io 게임 분석 (요구사항 1~7)

> 신뢰도 태그: **[확인됨]** 공개 자료 명시 · **[추정]** 역추론 · **[설계 결정]** 리부트에서 새로 정함
> 자료 출처 및 고지는 [README](./README.md) 참조.

원본 Vertix.io의 정체성을 한 줄로 요약하면:

> **3/4 탑다운 시점의 2D 아레나 슈터.** 좁은 미로형 맵에서 4분 라운드 동안,
> 매 리스폰마다 자유롭게 바꿀 수 있는 10개 클래스로 빠르게 교전한다.

이는 일반적인 1인칭 FPS와 본질적으로 다르며, 리부트에서 반드시 보존해야 할 핵심이다.

---

## 1. 핵심 게임 루프

### 1.1 매크로 루프 (라운드 단위) **[확인됨 + 추정]**

```
매치메이킹/입장
   ↓
클래스 선택  ──────────────┐
   ↓                       │
스폰 → 교전(이동/조준/사격) │  ← 사망 시 이 구간 반복
   ↓                       │
사망 → 리스폰 대기 ─────────┘  (리스폰 시 클래스 재선택 가능)
   ↓
라운드 종료 (4분 경과 또는 승리 조건 달성)
   ↓
결과 화면 / 코스메틱 보상 (로그인 유저)
   ↓
다음 라운드 (모드 투표)
```

- **라운드 길이: 약 4분** **[확인됨]** — 짧고 반복적인 세션이 핵심.
- **모드 투표:** 라운드 시작 시 3개 모드 후보 중 투표로 결정 **[확인됨]**.
- **라운드 종료 후 코스메틱 보상**(로그인 유저 한정) — 외형 아이템이 랜덤 지급 **[확인됨]**.

### 1.2 마이크로 루프 (초 단위 전투) **[추정]**

```
이동(WASD) ┐
조준(마우스)├→ 사격(좌클릭) → 적중/킬 또는 탄창 소진
보조무기   ┘                        ↓
                              재장전(R / 자동)
              ↓
   체력 관리: 힐팩 픽업 / 안전 지대로 후퇴
```

핵심 텐션: **좁은 통로 = 근접 교전 빈번**, 코너 싸움, 힐팩 쟁탈. 클래스마다 이 루프의 리듬이 다르다(저격수는 라인 유지, 화염방사병은 코너 매복 등).

### 1.3 리부트에서 보존해야 할 "게임성" 체크리스트

- [ ] **즉시 재진입성** — 죽어도 곧바로 다시 싸운다 (긴 대기 금지).
- [ ] **클래스 스위칭의 전술성** — 상황에 맞춰 매 리스폰마다 클래스 변경.
- [ ] **좁은 미로 맵의 코너 플레이** — 시야/사거리 싸움.
- [ ] **읽기 쉬운 탑다운 가독성** — 위에서 내려다보는 시점의 명료함.
- [ ] **4분 단위의 빠른 라운드 호흡.**

## 2. 클래스 시스템 분석

### 2.1 개요 **[확인됨]**

- 총 **10개 클래스**. 각 클래스는 **고유 외형 + 스탯 + 고정 로드아웃**을 가진다.
- **매 리스폰마다 자유롭게 클래스 교체 가능** — 이것이 핵심 전술 레버.
- 로드아웃은 **고정**(자유 조합 아님). 즉 "클래스 = 무기 + 체력 + 이동속도 패키지".
- 밸런스 철학: **고체력↔저기동 / 고데미지↔저지속** 의 트레이드오프. 일부는 **조크 캐릭터(Duck)** 로 의도적 비대칭.

### 2.2 클래스 표 **[확인됨: HP·무기 / 추정: 속도·역할]**

| 클래스 | HP | 주무기 | 보조/특이사항 | 이동속도(추정) | 역할 |
|--------|----:|--------|----------------|----------------|------|
| **Triggerman** | 100 | 머신건 (탄창 24, 25 dmg/발) | 올라운더, 기준점 클래스 | 보통 | 밸런스/입문 |
| **Hunter** | 50 | 스나이퍼 (100 dmg, 완전 정확) | 보조: 머신피스톨(탄창 5, 12 dmg, 고연사) | 보통 | 원거리 처형, 유리몸 |
| **Detective** | 100 | 데저트이글 (탄창 6, 50 dmg) | Hunter 즉사, 100HP 2발킬 | 보통 | 정밀 핸드캐넌 |
| **Run N Gun** | 100 | 토이 블라스터 (저데미지, 초고연사·초고속 재장전) | **최고 이동속도** | **빠름** | 기동/힐팩 선점 |
| **Vince** | 100 | 샷건 (펠릿 4개 산탄) | 근거리 폭딜 | 보통 | 근접 어쇼크 |
| **Rocketeer** | 120 | 로켓런처 (스플래시) | 예비 로켓 사실상 무한 | 느림(추정) | 광역/구역 통제 |
| **Spray N Pray** | 200 | 미니건 (최대 탄창, 최저 데미지/발) | 최고 체력·최저 기동 | **느림** | 탱커/지속화력 |
| **Arsonist** | 100 | 화염방사기 (초단거리·지속) | 좁은 통로에서 강력 | 보통 | 코너 매복 |
| **Nademan** | 110 | 네이드 런처 (반자동 유탄, 큰 탄창·약한 유탄) | 유일 무기(보조 없음 추정) | 보통 | 곡사/지역 거부 |
| **Duck** | **800** | 그라운드 파운드(근접 광역) | **조크 캐릭터**, 압도적 HP | 느림(추정) | 재미/난입 위협 |

> 수치 메모 **[확인됨]**:
> - Triggerman 머신건: 100HP를 **4발**, Nademan/Rocketeer를 **5발**, Hunter를 **2발**에 처치.
> - Hunter 스나이퍼: Triggerman/Detective/Arsonist/Vince를 **1발** 처치.
> - Detective 데저트이글: Hunter **즉사(1발)**, 100HP **2발**.

### 2.3 설계 시사점 **[설계 결정]**

- 클래스는 **데이터 주도(JSON/스키마)** 로 정의 — `{ hp, moveSpeed, weapons[], hitboxRadius }`.
- 무기·클래스를 분리하되, 로드아웃은 클래스에 고정 매핑.
- 밸런스 튜닝을 위해 모든 수치는 **서버 권위(authoritative)** 설정값으로 두고 핫리로드 가능하게.

## 3. 무기 시스템 분석

### 3.1 공통 규칙 **[확인됨]**

- 각 무기는 **유한한 탄창(mag size)** 을 가지며, 소진 시 **재장전** 필요.
- **예비 탄약은 사실상 무한** — 유탄/로켓/연료 포함. 즉 *탄창 관리는 있되 탄약 고갈은 없다.*
  → "재장전 타이밍"이 전술 요소이지, "탄약 절약"은 아님.

### 3.2 무기 아키타입 **[확인됨 수치 + 추정 분류]**

| 무기 | 발사 방식 | 탄창 | 발당 데미지 | 특성 |
|------|-----------|-----:|------------:|------|
| 머신건(Triggerman) | 자동 히트스캔 | 24 | 25 | 표준 DPS 기준 |
| 스나이퍼(Hunter) | 단발 히트스캔, 완전 정확 | (소) | 100 | 즉살, 저연사 |
| 머신피스톨(Hunter 보조) | 자동 히트스캔 | 5 | 12 | 근접 보조 |
| 데저트이글(Detective) | 반자동 히트스캔 | 6 | 50 | 고데미지 정밀 |
| 토이 블라스터(Run N Gun) | 자동(투사체?) | (중) | 저 | 초고연사·초고속 재장전 |
| 샷건(Vince) | 산탄 4펠릿 | (소) | 펠릿당 분산 | 근거리 폭딜 |
| 로켓런처(Rocketeer) | 투사체 + 스플래시 | (소) | 고(광역) | 자기 폭발 피해 가능성 |
| 미니건(Spray N Pray) | 자동 히트스캔 | 최대 | 최저 | 스핀업/지속화력 |
| 화염방사기(Arsonist) | 지속 원뿔, 초단거리 | 연료 | 틱 데미지 | 코너 제압 |
| 네이드 런처(Nademan) | 반자동 투사체(유탄) | 큰 | 약(광역) | 곡사/바운스 가능성 |
| 그라운드 파운드(Duck) | 근접 광역 슬램 | - | 고(광역) | 조크 무기 |

### 3.3 무기 메커니즘 분류 (구현 관점) **[설계 결정]**

발사 모델을 3종으로 추상화한다.

1. **히트스캔(Hitscan)** — 머신건, 스나이퍼, 데저트이글, 미니건, 머신피스톨, (샷건=다중 히트스캔 레이).
2. **투사체(Projectile)** — 로켓, 유탄, 토이 블라스터(추정). 이동·중력/바운스·충돌 시 폭발.
3. **지속 영역(Sustained/Cone)** — 화염방사기(원뿔 틱 데미지), 그라운드 파운드(원형 AoE).

무기 데이터 스키마 예시:

```jsonc
{
  "id": "machinegun",
  "type": "hitscan",        // hitscan | projectile | cone | aoe
  "damage": 25,
  "magSize": 24,
  "fireRateMs": 90,         // [추정]
  "reloadMs": 1500,         // [추정]
  "spreadDeg": 2,           // [추정]
  "rangePx": 600,           // [추정] 탑다운 좌표 기준
  "projectileSpeed": null,
  "splashRadius": null
}
```

> ⚠️ 연사속도·재장전·사거리·확산은 **모두 추정**. 원본 영상/코드로 캘리브레이션 필요.

## 4. 맵 구조 분석

### 4.1 확인된 특징 **[확인됨]**

- 모든 맵은 **좁은 통로와 날카로운 코너로 이루어진 미로(maze)**.
- **3/4 탑다운 시점** — 위에서 비스듬히 내려다보는 2D 렌더링.
- 맵 안에 **힐팩**과 (Lootcrate 모드의) **상자 스폰 지점**, **Hardpoint(중앙 거점)** 등 고정 오브젝트 존재.

### 4.2 역추론한 구조적 원칙 **[추정]**

좁은 미로 디자인은 다음을 강제하므로 의도적이다.

- **시야 차단** → 코너 플레이, 매복(특히 Arsonist/Vince/Shotgun류 근접 클래스 유리).
- **이동 동선의 병목** → 힐팩/거점 쟁탈, Rocketeer/Nademan의 곡사·광역이 통로에서 가치 상승.
- **저격 라인의 희소성** → Hunter는 긴 직선 통로/교차로에서만 강함 → 위치 선정이 핵심.

### 4.3 맵 데이터 모델 **[설계 결정]**

타일/폴리곤 기반 2D 충돌 맵으로 설계한다.

```
Map
├─ collision: 벽 폴리곤 또는 그리드 타일 (서버 권위 충돌)
├─ spawnPoints[]: 팀별/FFA 스폰 (적과의 거리 기반 안전 스폰 선택)
├─ healthPackSpawns[]: 힐팩 고정 위치 + 리스폰 타이머
├─ objectives:
│    ├─ hardpoint: 중앙 거점 영역 (Hardpoint 모드)
│    └─ lootcrateSpawns[]: 상자 스폰 후보 (Lootcrate 모드)
└─ meta: 권장 인원, 시야/렌더 레이어
```

- 충돌은 **타일 그리드(권장, 단순·결정론적)** 또는 폴리곤. MVP는 그리드 권장.
- 맵은 **데이터(JSON) + 에디터** 로 제작 → 콘텐츠 확장 용이.

## 5. 리스폰 시스템 추정 **[추정]**

자료에 "매 리스폰마다 클래스 교체 가능" 외 상세 수치가 없으므로 합리적으로 추정한다.

### 5.1 추정 메커니즘

- **사망 → 짧은 리스폰 딜레이(추정 2~3초)** → 리스폰. 빠른 재진입이 게임성의 핵심이므로 딜레이는 짧게.
- 리스폰 시 **클래스 재선택 UI** 표출(선택 안 하면 직전 클래스 유지).
- **스폰 지점 선택:** 적과 가장 가까운 스폰을 피해, 안전도가 높은 스폰 포인트에 배치(스폰킬 방지).
- **무적/페이드인:** 스폰 직후 짧은 무적(추정 0.5~1초) 또는 점멸로 즉사 방지.
- **팀 모드:** 팀 스폰 구역(또는 팀원 근처) 우선.

### 5.2 설계 결정 **[설계 결정]**

| 파라미터 | 제안 기본값 | 비고 |
|----------|------------|------|
| 리스폰 딜레이 | 2.0s | 모드별 조정 가능 |
| 스폰 무적 | 0.75s | 이동/사격 시 즉시 해제 옵션 |
| 스폰 선택 | "가장 안전한 스폰" 점수화 | 적 거리·시야 기반 |
| 클래스 변경 | 리스폰 화면에서만 | 라이브 중 변경 불가 |

## 6. 힐팩 시스템 추정 **[추정]**

### 6.1 확인/추정

- **[확인됨]** 맵에 **메드팩(힐팩)** 이 존재하며, Run N Gun이 속도로 **힐팩을 선점**한다는 언급이 있음.
- 즉 **체력 재생은 자동 리젠이 아니라 맵 픽업 중심**일 가능성이 높음 → **힐팩 쟁탈이 핵심 자원 게임**.

### 6.2 추정 메커니즘 **[추정]**

- 힐팩은 **맵 고정 위치**에 스폰, **픽업 시 즉시 회복**.
- 픽업 후 해당 지점은 **쿨다운(추정 10~20초) 후 재스폰**.
- 회복량: **정량(추정 +50 HP)** 또는 **풀 회복** 중 택1. 좁은 맵·빠른 전투 특성상 정량 회복이 쟁탈 긴장감↑.
- **자동 체력 재생은 없음(추정)** 또는 매우 느림 → 힐팩 의존도 유지.

### 6.3 설계 결정 **[설계 결정]**

| 파라미터 | 제안 기본값 | 비고 |
|----------|------------|------|
| 회복량 | +50 HP (최대치까지) | Spray/Duck 등 고HP는 상대적으로 약하게 작용 |
| 픽업 방식 | 접촉 즉시 | 채널링 없음 |
| 재스폰 쿨다운 | 15s | 맵별 조정 |
| 패시브 리젠 | 없음 (또는 5초 무피해 후 소량) | 원본 느낌 우선 시 "없음" |

## 7. 점수 시스템 추정 **[확인됨 일부 + 추정]**

### 7.1 확인된 사실 **[확인됨]**

- **FFA:** 가장 먼저 **1500점** 도달 시 승리. **보통 15킬** → 즉 **킬당 100점** 환산.
- **TDM:** 두 팀이 각자 **진행 바**를 채우며, 팀원들의 점수(킬)로 바가 차고 먼저 가득 차면 승리.
- **Hardpoint:** 중앙 거점에 서 있는 **1초당 10점**(개인·팀 동시 적립). Hunter가 거점 위 적을 저격하면 점수 견제.
- **Lootcrate:** 맵의 정해진 위치에 스폰되는 **상자 수집**으로 점수 획득.
- **Boss Hunt:** 보스는 **20킬** 시 승리, 일반 플레이어는 **보스 처치** 시 승리.

### 7.4 설계 결정 **[설계 결정]**

- 점수는 **서버 권위**로 집계. 이벤트 기반(`onKill`, `onCaptureTick`, `onPickup`)으로 누적.
- 라운드 종료는 **(목표 점수 도달) OR (시간 종료)** 의 단일 종료 판정기로 통합.
- 모드는 **규칙 플러그인(win condition + scoring rule)** 으로 추상화하여 모드 추가를 쉽게.
